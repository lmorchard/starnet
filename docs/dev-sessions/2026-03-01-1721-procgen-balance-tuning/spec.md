# Spec: Procgen Balance Tuning

## Goal

Build measurement tooling to analyze the procedural generator's output across
the full 6×6 difficulty matrix, then use those measurements to make targeted
balance adjustments. The tooling should produce LLM-legible reports that Claude
can read and reason about for future tuning passes.

---

## Background

The generator has 36 difficulty combinations (F–S × F–S) but only F/F and B/B
have been playtested. Session 1 flagged several issues:

- F/F is too thin (7 nodes, invisible ICE)
- B/B cash pressure is tight
- Card count (6) doesn't scale with difficulty
- S/S is likely uncompletable
- Careless-user set piece has never been observed in play

Before making tuning changes, we need data across the full matrix to understand
what the generator actually produces and what the player experience would be at
each difficulty level.

---

## Phase 1: Network Census Tool

### What it is

A batch script (`scripts/network-census.js`) that generates networks across all
36 difficulty combinations, collects metrics, and prints an LLM-readable report.

### Sampling

10 seeds per difficulty combination (360 total generations). Fixed seed set
(`census-0` through `census-9`) so results are reproducible across runs. Report
shows min/avg/max for each metric per combo.

### Topology metrics (per generation)

Measured directly from generator output — no game state needed:

| Metric | Source |
|--------|--------|
| Total node count | `network.nodes.length` |
| Node count by type | Group by `node.type` |
| Critical path length | BFS shortest path from `startNode` to nearest lootable target |
| Gate count on critical path | Firewalls on the BFS path |
| Node grades on critical path | Ordered list of grades the player must crack |
| Set piece fired | Whether careless-user (or any piece) injected nodes |
| ICE grade | From `network.ice.grade` |
| Depth spread | Max depth layer present |

### Resource estimates (derived from topology + combat math)

These are statistical expectations, not exact gameplay outcomes. Assumptions:

- **Exploits to own a node**: 2 successful exploits (locked → compromised → owned)
- **Success probability**: `min(0.95, quality × gradeModifier + matchBonus)`
  - `matchBonus`: +0.40 (assume player probes first and has a matching card)
  - `quality`: use average for the starting hand's rarity mix
    - Common avg: 0.375, Uncommon avg: 0.60, Rare avg: 0.825
    - Weighted average based on `HAND_BUDGET[moneyCost]`
  - `gradeModifier`: from `GRADE_MODIFIERS` (F=0.90 down to S=0.05)
- **Expected card uses per node**: `2 / successProbability`
- **Expected total card uses on critical path**: sum across all critical path nodes (excluding gateway/wan)
- **Starting card uses available**: sum of uses across starting hand (Common=3, Uncommon=5, Rare=8)
- **Card deficit**: expected uses needed minus starting uses available (positive = need darknet store)
- **Estimated darknet cost**: card deficit × average card price (if darknet store pricing is available)

The "matched exploit" assumption represents skilled play: probe first, buy
matching cards from the darknet store when needed. This is the play pattern the
difficulty curve should be designed around. Lower grades are accessible enough
that even suboptimal play works; higher grades reward players who have learned
the probe-then-match loop.

### Report format

Text-based, structured, designed for LLM readability:

```
=== NETWORK CENSUS REPORT ===
Seeds: census-0 through census-9 (10 per combo)
Date: 2026-03-01

--- SUMMARY TABLE ---
TC  MC  Nodes  CritPath  Gates  GradeSpread    EstUses  StartUses  Deficit  SetPiece%
F   F   7.0    2.0       0.0    F,F            4.3      27.0       -22.7    0%
F   D   7.0    2.0       0.0    F→D,D          5.1      27.0       -21.9    0%
...
S   S   14.2   5.3       3.0    B→A→S→S→A      38.4     43.0       -4.6     60%

--- DETAIL: B/B (10 seeds) ---
Seed       Nodes  Path  Gates  Grades         EstUses  Piece?
census-0   11     3     1      C,B,A          16.2     no
census-1   11     3     1      C,A,B          17.8     yes
...

--- FINDINGS ---
(Auto-generated observations about outliers, concerning combos, etc.)
```

The FINDINGS section can be either auto-generated by simple heuristics (e.g.
"deficit > 0 in X combos") or left for Claude to fill in when reading the report.

### Usage

```bash
node scripts/network-census.js                    # full 36-combo summary
node scripts/network-census.js --detail B,B       # detailed view for one combo
node scripts/network-census.js --detail all       # detailed view for every combo
node scripts/network-census.js --seeds 20         # override sample count
```

## Phase 2: Balance Tuning

### Process

1. Run the census tool and review the report
2. Identify combos that are obviously broken (impossible, trivial, or degenerate)
3. Make targeted changes to budget tables and layer definitions
4. Re-run the census and compare before/after
5. Repeat until the difficulty curve looks reasonable

### Known candidate changes (from session 1 notes + assessment)

These are starting points, not commitments — the census data may reveal different
priorities:

- **Minimum node count floor**: F/F at 7 nodes feels too sparse. Consider adding
  extra filler workstations at low difficulty so there's something to explore.
- **Hand size scaling**: `HAND_BUDGET` scales rarity but not count. At A+ difficulty,
  dealing 7–8 cards instead of 6 would offset higher failure rates.
- **Filler count decoupled from routers**: currently `count = state.routing.length`.
  A floor of 2 or a formula like `max(2, routing.length)` would fill out sparse
  networks.
- **Starting cash at B+**: session 1 noted ¥1,000 felt tight (may be stale — current
  `CASH_BUDGET[B]` is ¥1,500). Census data will show whether the current values
  are adequate.

### Success criteria for tuning

No formal difficulty targets — this is exploratory. But some rough guidelines:

- F/F should be completable with the starting hand and no darknet purchases
- C/C should require some card management but not feel punishing
- S/S should be very hard but theoretically possible with optimal play + darknet
- Card deficit should scale smoothly across the matrix, not jump wildly between
  adjacent grades
- No combo should produce degenerate topology (e.g. 3 nodes, or 20+ nodes)

---

## Phase 3: Set Piece Test Coverage

Add a snapshot test with `forcePieces: ["careless-user"]` to catch regressions in
the set piece path. This is a small addition but closes a real coverage gap —
currently zero tests exercise set pieces through the generator.

---

## Out of Scope

- Exploit success rate tuning (combat math changes) — separate session if needed
- New set pieces — the census may suggest ideas but implementation is future work
- Automated bot playthroughs — worth exploring after we have census data to inform the design
- Loot/reward scaling with difficulty — game state layer concern, not generator
- Interactive/REPL-style explorer — natural follow-up once batch tooling exists
